深入理解Java虚拟机

发表于 2017-06-30 | 分类于 读书笔记

————Undersatanding the JVM Advanced Features and Best Practices, 一本理解JVM的必读经典

走进Java

  • JVM发展史，各家JVM实现细节并不一样，JVM规范只制定了核心内容，各家根据情况实现了自己的JVM。
  • 探索和理解JVM避免不了阅读源代码，Oracle收购Sun后的JDK不再开源，开源版本在OpenJDK上继续迭代，编译配置好一套OpenJDK调试环境尤为重要。

自动内存管理机制

内存区域划分

Program Counter Register(程序计数器)、Method Area(方法区)、VM Stack(虚拟机栈)、Heap(堆)、Native Method Stack(本地方法栈):

  • 程序计数器 是所有机器指令执行逻辑必要的一个组成，记录程序执行指令的位置。JVM里每条线程有自己独立的程序计数器，切换线程时能快速恢复现场。
  • 方法区 用于存储编译后的类信息、常量、静态变量等数据，有的JVM如HotSpot将其实现为容量不可扩展的永久代，这使得代码容易超出上限而溢出，尤其是使用String.intern()这种可能持续扩充方法区的逻辑。[TIPS]: 深入解析String#intern。进一步，有的虚拟机会划分出运行时常量池用来放置String常量，JDK7开始甚至把这块挪到了堆里，以供方便的扩充。
  • 虚拟机栈 程序执行过程中，每次方法调用对应着一个栈帧的入栈出栈，记录着方法的局部变量、操作数栈、方法出口等信息。
  • 堆 所有代码逻辑里建立的对象实例都得在堆上申请分配内存，通常所说的垃圾回收也主要是针对堆里内存的回收。
  • 本地方法栈 调用操作系统本地方法的栈区，HotSpot索性将其和虚拟机栈区放在一起管理。

需要了解各区域内存异常的情况以及应对方案。

垃圾收集器与内存分配策略

Garbage Collection需要完成三件事情：那些内存需要回收、什么时候回收、怎么回收：

  • 那些需要回收 引用计数法(无法解决循环引用)，可达性分析(GC Roots)
  • 什么时候回收 程序在安全点上会有OopMap加快GCRoots枚举，安全区是为了避免GC等待挂起的线程到达安全点(安全区内对象引用关系不会发生改变)
  • 怎么回收 Mark-Sweep, Copying(Eden\Survivor), Mark-Compact(老年代)

可以确定的是，现代收集器都是用了分代收集的策略，根据对象不同的特点进行区分对待。新生代的对象经过种种策略会转移到老年代。随着JVM的演化，垃圾收集器也有着一系列的演化。演化的核心点在于提高并发、减少停顿时间、提高吞吐率。Serial、Parallel、CMS、G1，直到Java9，G1才是默认的垃圾收集器。

垃圾收集器的使用需要结合具体场景分析GC日志具体调优

虚拟机执行子系统

类文件结构

java虚拟机在操作系统之上构建了一个统一的平台，基于之上的各个语言及其编译器均通过生成字节码在虚拟机中运行。类文件结构说的就是字节码文件.class的结构，可想而知就是一些固定规则的结构，可以让虚拟机拿来直接执行的虚拟机指令。

类加载机制

虚拟机在必要时需要把类的字节码加载到内存，并对数据进行验证、准备、解析和初始化。虚拟机会尽可能晚的去加载一个类，毕竟不到用时就加载会浪费空间。

类加载器实现了通过一个类的权限定名获取其二进制字节流。虚拟机只定义了基础的启动类加载器(Bootstrap ClassLoader)，其他类加载器均在虚拟机外部由Java语言实现，包括扩展类加载器、应用程序类加载器和用户自定义加载器。这些加载器以双亲委派模型配合加载类(TIPS: 在工程中自定义个Object，使用java编译运行不会有问题)。如果使用Tomcat/Rersin容器来启动Java服务，容器可能定义了自己的一套加载器，用于方便的隔离和共享APP的类库。TIPS: 另外，类似JDBC这种SPI(Service Provider Interface)性质的类加载，会通过线程上下文类加载器打破双亲委派模型。

毕竟双亲委派这种简单的模型很难满足现实复杂的需求，后续发展的OSGi模块规范，以及JDK9的jigsaw模块管理，都为复杂的问题提供解决方案。

虚拟机字节码执行引擎

执行引擎决定着如何将字节码一步步执行下去，最直接的类似计算机体系结构，通过栈帧、操作数栈、方法调用等一步步执行。

其中方法调用有的在编译器就确定了调用地址(重载静态分派)，有的需要在运行时决定(重写动态分派，类似虚函数表的实现)。为了实现动态语言的一些特性，JDK1.7之后加了新的动态调用指令，可以由用户在代码逻辑中动态分派。

字节码执行可能是解释执行，也可能是通过JIT即时编译生成机器代码再执行。

程序编译与代码优化

早期(编译期)优化

javac的编译期优化比较少，主要是词法语法分析，注解的处理，泛型等一些语法糖的处理。

晚期(运行期)优化

运行期的优化才是提现整个虚拟机技术的核心内容。HotSpot虚拟机通过解析器和C/S(Client, Server)编译器的配合，来动态调整执行过程，主要对方法和循环体调用次数的监控来触发JIT编译。优化的手段非常之多，包括数组边界检查消除、方法内联(这里方法内联不仅仅是简单的编译期确定的方法内联，包括动态分派方法的内联，使用了一些复杂的分析技术)、逃逸分析(主要确定对象作用于，进而采取更高效的优化)。

高效并发

Java内存模型与线程

volatile关键字与指令重排

原子性、可见性与有序性，Happens-Before原则(线程内部的顺序、对象锁的释放与获取、volatile变量的赋值与读取、传递性)

Java内线程与系统线程是1:1关系，由CPU统一调度。

线程安全与锁优化

线程安全: 互斥同步(悲观策略)、非阻塞同步(乐观策略CAS)、无同步方案。

锁优化是JDK1.5到JDK1.6一个重要改进，实现了适应性自旋、锁消除、锁粗化、轻量级锁、偏向锁等等
